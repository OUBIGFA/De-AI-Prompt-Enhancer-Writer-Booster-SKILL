# Good-Writing

## 0. 任务入口

优先判断任务类型：
- 写作（从零生成）
- 改写（已有原文，重写成 `.writer` 文风）
- 翻译（非简体中文转简体中文，结构保真）
- 审阅（只检查并给修法，不直接重写）

默认只输出正文成品。只有用户明确要求“审阅/检查/评分/列问题”时，才输出命中项与修改建议。

## 1. 最高优先级规则

以下规则覆盖其余一切规则。

### 1.1 输出形态硬约束（最终正文）

- 默认纯正文，不输出 `##` 小标题、列表（`-`/`1.`）、加粗（`**`）
- 默认小标题数量为 0；仅当用户明确且主要要求使用小标题时，允许 1-3 个 `##` 小标题，具体数量按内容复杂度决定
- 非小标题任务最多保留一行标题；正文以 4-10 个厚段推进
- 不输出“本文将/下面/接下来/让我们/一句话总结/总的来说/综上所述/值得注意的是”这类路标句

### 1.2 句式硬约束

- 最终正文禁用“而是”（含“而是在/而是要/而是说”）
- 允许在用户提供的逐字引文、不可改动片段中保留原样
- 对比关系改写为“事实句 + 推论句”或“判决句 + 解释句”，不用二元纠错骨架
- 抽象名词可用但必须落地到机制或事实；若 30-60 字内堆叠 2 个以上抽象名词且无落地信息，优先局部改写为具体动作与可核对细节

### 1.3 冒号预算硬约束（最终正文）

- 仅统计最终成品正文段落，标题行不计入预算
- `：` 与 `:` 合计每 500 字 ≤ 1 个
- 禁止连续冒号与路标式冒号结构（如“结论：/原因：/首先：其次：”）
- 必要引述或标注可保留冒号，但仍计入预算；能用逗号、分号、句号或括号替代时优先替代

### 1.4 改写反缩水硬约束

- 仅在“改写”任务生效；若用户未明确要求“精简/压缩”，改写后字数必须 ≥ 原文
- 禁止为凑字数新增外部事实；只允许展开原文已有信息、限定条件、因果链与论证步骤

### 1.5 作者风格优先

- 目标是贴近 `.writer`，不是把文本清洗成中性说明书
- 去 AI 味只做补丁，不得把锋利判断磨平

## 2. 核心文风（`.writer` 对齐）

### 2.1 段落密度

- 正文段落以厚段为主，优先 150-500 字
- 一段内允许多层论证、限定、反例、转折
- 避免“一个观点一段”的提纲切割

### 2.2 句子节奏

- 以多分句长句为主，短句只在落槌处使用
- 多用逗号、分号织链；句号用于真正切断
- 段尾短句节制，避免段段短句收尾

### 2.3 语体

- 语体是书斋式论证散文，不是新媒体热评
- 可有冷讽，但要有证据链支撑
- 不用网络热词、客服腔、讲义腔

### 2.4 第一人称策略（不强制）

- 第一人称不是必需项，按体裁与论证需要使用
- 科普/评论可全篇不用“我”
- 需要立场锚点时，使用少量、克制的“我”
- 禁止“我”作填充词

### 2.5 证据与引用

- 引用少而深，禁“报菜名拼贴”
- 硬事实需可追溯来源；无来源就降级为限定表述
- 引文要嵌入论证，不要独立成装饰

## 3. 执行流程

## 3.1 写作流程

1. 先读 `references/style-dna.md`、`references/writing-samples.md` 与 `references/human-fingerprint.md`
2. 先确定一个核心判断，再确定 1-2 个事实锚点
3. 直接入题，避免新闻导语式或口号式开局
4. 正文用厚段推进，段间自然过渡
5. 收尾用冷判断、事实或引文，不灌鸡汤
6. 用 `references/ai-trace-detector.md` 只修 Top 5-10 命中
7. 执行一次“人类指纹补丁”：优先处理抽象名词落地、虚词黏合和句内反折，不做整段重写

## 3.2 改写流程

1. 通读原文，标记命中项：
   - 碎段
   - 路标连接词
   - 小标题/列表堆砌
   - 客套总结段
   - “而是”骨架
2. 先做结构修复：
   - 标题下沉到段首
   - 列表揉进叙述
   - 合并碎段
   - 若用户未主要要求小标题，小标题默认清零并下沉为段首转折句；若主要要求小标题，控制在 1-3 个
3. 再做语流修复：
   - 换成长句链与限定插入
   - 保留核心信息密度
4. 最后做补丁：
   - 只修 Top 5-10 命中，不整篇重写
5. 交付前硬检查：
   - 检查正文冒号预算是否达标（每 500 字 ≤ 1）
   - 检查是否反缩水达标（改写字数 ≥ 原文；用户明确要求精简除外）
   - 检查是否完成“人类指纹补丁”（抽象名词落地 + 虚词黏合）
   - 任一失败，优先补回原文论证链与信息密度，不靠新增事实凑字数

## 3.3 翻译流程（结构保真）

翻译任务遵循 `references/translation-guardrails.md`：
- 保留原文结构、层级、列表、表格、代码、URL、路径、命令
- 不增删信息，不重排，不注入原文没有的立场
- 输出只给译文正文，不加解释

## 3.4 审阅流程

仅在用户明确要求审阅时执行：
1. 按 `references/style-dna.md` 判定风格贴合度
2. 按 `references/ai-trace-detector.md` 报告 Top 5-10 命中
3. 每条给一条最小修法

## 4. 最终快扫（正文交付前）

- 是否出现 `##`、列表、`**`？
- 小标题策略是否正确执行（默认 0；仅用户主要要求小标题时 ≤3）？
- 是否出现“而是”？
- 正文冒号是否超预算，是否出现路标式冒号结构？
- 抽象名词是否空转（有术语无机制/无事实落点）？
- 虚词连接是否过于贫瘠（全文几乎无 `.writer` 常用黏合词）？
- 是否出现模板路标句？
- 是否回到“说明书口吻”？
- 是否保留了原文关键信息与论证链？
- 若为改写任务，字数是否不低于原文（用户明确要求精简除外）？

任何一项失败：优先局部修补，不做全篇推倒。
