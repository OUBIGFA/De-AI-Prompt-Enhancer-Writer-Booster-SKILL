- 结构保真：段落/标题/列表/引用/表格/缩进/空行保持一致
- 一比一对应：输出每个段落/列表项/表格单元都能对应回源文
- 不可改动片段原样复制（代码、URL、路径、命令、占位符等）

## 2) 结构保真（必须遵守）

- 不合并、不拆分段落，不调整段落顺序
- 标题层级与数量保持一致（`#`/`##`/`###`... 不变）
- 列表结构保持一致：层级、编号、项目数不变
- 引用块结构保持一致（`>` 层级与换行不变）
- 表格结构保持一致：列数、分隔线、对齐标记不变
- 空行、缩进、刻意换行必须保留
- Markdown 链接 `[可见文字](URL)`：只处理可见文字，`URL` 不改

## 3) 不可改动片段（原样复制，禁止改写/翻译）

以下内容出现在哪里，就在哪里原样保留：

- fenced code block：```...```（含语言标注与内部全部内容）
- 缩进代码块（每行以空格/Tab 开头）
- 内联代码：`` `...` ``
- LaTeX/数学公式：`$...$`、`$$...$$`
- URL、email
- 文件路径、包名、类名、函数名、命令与参数（含 `--flags`）
- 配置键名、占位符与模板语法：`${...}`、`{{...}}`、`%s`、`{0}` 等
- 版本号、ID、哈希、精确数字与单位
- 任何无法确认可安全翻译的强专业片段（宁可保留）

## 4) 英文与专名策略（折中）

默认不再执行“含拉丁字母即保留”，改为“专名与技术 token 保护，普通词翻译”：

- 必须保留英文原文：
  - 人名、品牌名、产品名、组织名、项目名
  - 文件名、类名、函数名、变量名、命令、参数、路径、URL、占位符
  - 代码块与内联代码中的全部内容
- 默认翻译为中文：普通英文词与常见短语（如 assistant、employee、framework、onboarding）
- 如中文化会损失技术精度，可采用“中文 + 英文括号保留一次”
- 用户明确要求某词必须保留或必须翻译时，按用户指令覆盖

## 5) 信息守恒（翻译与改写共同约束）

- 禁止新增：背景说明、例子、结论、评价、立场、推断
- 禁止删除：任何承载信息的事实、条件、限定、语气强度
- 禁止把原文“总结化”替代正文
- 禁止将原文论证顺序改为另一套逻辑链

## 6) 本地化改写规则（中等强度，段内执行）

保真改写允许在段内进行中等强度本地化，以适配中文语境；但段落和列表结构不变。

允许：

- 句子合并/拆分、语序重排、连接词替换
- 英文短句串改写为中文常见转折/递进句式
- 去除不承载信息的语气词/填充词（仅在语境确认时）
- 去翻译腔/去 AI 腔（只改表达，不改信息点）

禁止：

- 扩写、缩写、重组信息块
- 合并或拆分段落/列表项
- 引入源文不存在的事实或判断

## 7) 去噪（仅对口语转写类输入生效）

当且仅当源文明显是口语/访谈/会议纪要/聊天记录时：

- 删除不承载信息的语气助词、犹豫词、附和词
- 对断续碎碎念可做简洁浓缩，但不得误删实质信息
- 黑名单（仅在不承载信息时可删）：`嗯`、`哦`、`呃`、`这个`、`那个`、`实际上`、`就是说`、`Yeah`、`OK`、`Ok`、`Nice`、`Cool`、`Uh`、`Um`、`Like`、`You know`、`Sort of`、`I mean`
- 语境保留：
  - `cool` 表示物理状态时保留语义（如“低温/冷却”）
  - `OK/Ok` 作为界面术语时保留界面语义（如“Click OK”）

## 8) 专业符号化与单位规范（仅限自然语言区域）

以下规则只在自然语言区域生效，不进入不可改动片段：

- 负数：`Negative 50` -> `-50`
- 角度：`360 degrees` -> `360°`
- 百分比：`20 percent` -> `20%`
- 乘法/尺寸：`10 by 10` -> `10×10`
- 范围/误差：`from 10 to 20` -> `10~20`；`plus or minus 5` -> `±5`
- 保留或使用 `>`、`<`、`=` 等符号
- 单位使用标准符号：`kg`、`m`、`V`、`Hz`；默认不做单位换算
- 数字与单位符号之间不得加空格（如 `50kg`、`500nits`）

## 9) 中西文混排（仅限自然语言区域）

- 中文与数字/英文单词之间加一个半角空格
- 例外：数字与 `%`、`°`、单位符号之间不加空格（如 `100%`、`360°`、`500nits`）
- 不在 Markdown 语法关键位与不可改动片段内强行加空格

## 10) 输出约束

- 只输出目标文本，不附加译者注、分析、总结、解释
- 默认目标语言为简体中文（除非用户明确指定其他目标语言）
- 默认执行两遍法：
  - Pass 1：逐段直译，先确保信息点完整
  - Pass 2：段内本地化改写，让中文更自然（不增删信息点）
- 若用户明确要求自由改写/允许增删，本护栏不生效；否则默认强制生效

## 4) 专有名词策略（默认：常见即翻译）

- 有稳定中文译名的常见机构/地名/条目：使用常见译名
- 其余专名：保留原文拼写，不硬造译名
- 人名：默认保留原文拼写；仅在中文语境中译名极稳定的历史名人可用常见译名；不确定就保留原文

## 5) 允许的轻度风格化（仅限“翻译更像人话”）

- 允许把直译腔改成更自然的简体中文（例如避免硬直译语序），但不得改变语义、信息量与语气强度
- 不添加原文没有的立场、类比、第一人称或额外攻击性；不把翻译变成“改写”

## 6) 严禁事项

- 不加“译者注/补充说明/背景解释/总结”
- 不删除重复、不修饰立场、不改变语气强度
- 不把原文的列表、表格、标题“散文化重写”

## 7) 译后优化（两遍法：快译 → 精修）

翻译建议固定用“两遍法”，先快后准，最后再把中文磨到自然：

- Pass 1（快译）：按原文顺序逐段逐行译完全文（结构保真、不可改动片段原样保留），不要在半路反复打磨个别句子
- Pass 2（精修）：在**不改结构/不增删信息/不改语气强度**的前提下，系统清理翻译腔与排版噪音

Pass 2 只能作用于自然语言区；代码块/内联代码、URL、路径、命令、占位符、版本号/ID/哈希/精确数字与单位等均按 §3 原样保留。

- 去翻译腔（优先清理直译框架，不改信息量）：如“当……时”“在……的情况下”“对于……来说”“……之一”“进行……”“以……为基础/为前提”等

在教程、配置说明、参数解释、数据/范围/维度描述等“技术/说明类”文本中，优先用符号替代啰嗦文字（不改变语义）：

- 比较与逻辑：保留/使用 `>`、`<`、`=` 等符号表达

### 7.3 单位写法标准化（默认不做单位换算）

- 单位用标准符号表示，避免中文全称：`50kg`、`100m`、`220V`、`50Hz`
- 不做单位换算；只做写法规范化（例如 `meters` → `m`）
- 冲突裁决：**数字与单位符号之间严禁加空格**

### 7.4 中西文混排（盘古之白）

- 不在 Markdown 语法关键位置与不可改动片段内强行加空格（尤其是链接 `URL`、代码、命令、路径等）
